Common Coding Mistakes.

This document presents the most common mistakes and problems that users encounter when writing their custom formulas. Please read carefully to avoid making similar errors.

1. BarCount vs BarIndex()

THEORETICAL BACKGROUND:

There is a fundamental difference between BarCount and BarIndex(). BarCount is a NUMERIC variable that holds just one NUMBER (the count of elements in an array).

BarIndex() is a FUNCTION that returns an ARRAY representing a bar index.

BarCount is intended to be used in FOR loops. That's why it holds the actual number of array elements so you can iterate as follows:

for( i = 0; i < BarCount; i++ )

BarIndex() on the other hand is a faster replacement of (Cum(1)-1).

LastValue( BarIndex()) does not give you the number of array elements.

There is the following relation between those two:

bi = BarIndex();

BarCount == 1 + LastValue(bi)-bi[0]; 

EXAMPLE 1:

for (i=0;i < BarCount;i++)
{
 A1=Close[ BarIndex() ]; // WRONG !!
}

BarIndex() returns an ARRAY, while all that is needed is a NUMBER of the current Close array element. Due to the fact that i increases by 1 in each iteration, the correct formula should be:

for (i=0;i < BarCount;i++)
{
 A1=Close[ i ]; // CORRECT
}

EXAMPLE 2:

for (i=0; i<=BarIndex();i++ ) // WRONG
{
  //formula
}

It’s not allowed to use an ARRAY inside a for loop, and BarIndex() returns an ARRAY. That is why it’s necessary to change the formula above to:

for (i=0; i< BarCount ;i++ )  // CORRECT
{
  //formula
}

  1. IIf vs if-else

THEORETICAL BACKGROUND:

The IIf( ) function is used to create conditional assignments. It returns an ARRAY.

variable= IIf( EXPRESSION, TRUE_PART, FALSE_PART );

The above "IIf" statement means: For each bar EXPRESSION is true, assign TRUE_PART to the variable, otherwise (when EXPRESSION is false) assign FALSE_PART.

The if keyword executes statement1 if expression is true (nonzero); if 'else' is present and expression is false (zero), it executes statement2. After executing statement1 or statement2, control passes to the next statement. Expression must be boolean ( True/False) type (so it CANNOT be an ARRAY because there would be no way to decide whether to execute statement1 or not, if for example, an array was: [True,True,False,.....,False,True] )

if( expression )
   statement1
else
   statement2

EXAMPLE 1:

IIf( condition, result = 7, result = 9 ); WRONG

result = IIf( condition, 7, 9 ); CORRECT

EXAMPLE 2:

variable = IIf(Condition, "Text 1","Text 2" ); WRONG

IIf( ) function returns an array, NOT STRING, so it’s impossible to assign text to a variable with the use of IIf. Use WriteIf( ) function instead:

variable = WriteIf( condition, "Text 1", "Text 2" ); CORRECT

EXAMPLE 3:

if( Close > Open )   — WRONG
  Color = colorGreen; //statement 1
else
  Color = colorRed; //statement 2

Plot(Close,"Colored Price",Color,styleCandle);

The above example is wrong, as both Open and Close are arrays and an expression such as Close > Open is also an ARRAY. The solution depends on the statement. It's either possible to implement it on a bar-by-bar basis, with the use of a FOR loop:

for(i=0;i<BarCount;i++)
{
   if( Close[ i ] > Open[ i ] )       — CORRECT
         Color[ i ] = colorGreen;
   else
         Color[ i ] = colorRed;
}

Plot(Close,"Colored Price",Color,styleCandle);

It is also possible in this case to use IIf( ) function:

Color = IIf(Close > Open, colorGreen, colorRed);  — CORRECT
Plot(Close,"Colored Price",Color,styleCandle);

  1. ‘=’ vs ‘==’

THEORETICAL BACKGROUND:

‘=’ is a variable assignment operator

‘==’ is an equality check

EXAMPLE:

result=IIf(Variable=10,True_part,False_part); WRONG

If you want to check if variable is equal to 10, you MUST use "=="

result=IIf(Variable==10,True_part,False_part); CORRECT

  1. TimeFrameSet(), TimeFrameRestore( ), TimeFrameExpand( )

THEORETICAL BACKGROUND:

The TimeFrameSet( ) replaces current price/volume arrays: open, high, low, close, volume, openint, avg with time-compressed bars of a specified interval. Once you switch to a different time frame, all calculations and built-in indicators operate on the selected time frame. To get back to the original interval, call TimeFrameRestore( ) function. The TimeFrameExpand( ) is used to decompress array variables that were created in a different time frame. Decompressing is required to properly display and use the array created in a different time frame.

EXAMPLE 1:

TimeFrameSet(inWeekly);
MA14_Weekly=MA(Close,14);
TimeFrameRestore();
Buy=Cross(Close,MA14_Weekly); — WRONG

The above formula is wrong, as the MA14_Weekly variable should be EXPANDED to match the original timeframe. The correct code should be:

TimeFrameSet(inWeekly);
MA14_Weekly=MA(Close,14);
TimeFrameRestore();
Buy=Cross(Close,TimeFrameExpand(MA14_Weekly,inWeekly) ); — CORRECT

EXAMPLE 2:

“I would like to display weekly Close prices and I use the following formula, but despite the fact I've set weekly timeframe, it seems to show daily dates. Why?”

TimeFrameSet(inWeekly);
Filter=1;
AddColumn(C,"Close");

TimeFrameSet( ) function replaces the original Close array with weekly values, but leaves the original timestamp. Time frame functions are generally necessary when you want to use more than one timeframe within the formula and in the above case are not necessary. It’s enough to set Periodicity to WEEKLY (Analysis -> Automatic Analysis -> Settings -> General tab ).

EXAMPLE 3:

TimeFrameSet(inWeekly);
MA14_Weekly=MA(Close,14);
TimeFrameRestore();
Buy=Cross(Close,TimeFrameExpand(MA14_Weekly,inDaily) ); — WRONG

It's always necessary to indicate in the TimeFrameExpand( ) function, which timeframe was the variable calculated in. So if MA14_Weekly was calculated out of weekly data, inWeekly should be the correct parameter of the TimeFrameExpand( ) function.

TimeFrameSet(inWeekly);
MA14_Weekly=MA(Close,14);
TimeFrameRestore();
Buy=Cross(Close,TimeFrameExpand(MA14_Weekly,inWeekly) ); — CORRECT

EXAMPLE 4:

“I have a database with `base time interval` equal to 1_Minute. I set periodicity to 15-Minutes and use the TimeFrameSet( ) function to get values for a 10-Minute Moving Average. I want to check for the crossover of a 10-Min MA calculated out of 1-Minute bars and a 10-period MA calculated from 15-Minute bars. Why doesn't the formula below work properly?

TimeFrameSet(in1Minute);
MA10_Min=MA(Close,10);
TimeFrameRestore();
Buy=Cross(TimeFrameExpand(MA10_Min,in1Minute),MA(Close,10) );

You can only compress data from shorter interval to longer interval. So when working with 1-minute data you can compress to 2, 3, 4, 5, 6, ....N-minute data. But when working with 15 minute data you cannot get 1-minute data bars. In a similar way if you have only EOD data you cannot access intraday time frames.

The solution is to set periodicity to 1-Minute, and calculate the 10-period MA calculated from 15-Minute bars with the use of the TimeFrameSet( ) function:

TimeFrameSet(in15Minute);
MA10_15Min=MA(Close,10);
TimeFrameRestore();

Buy=Cross( MA(Close,5), TimeFrameExpand(MA10_15Min, in15Minute) );

  1. WriteIf( )

THEORETICAL BACKGROUND:

writeif( EXPRESSION, "TRUE TEXT", "FALSE TEXT" )

If EXPRESSION evaluates to "true", then the TRUE TEXT string is displayed within the commentary. If EXPRESSION evaluates to "false", then the FALSE TEXT string is displayed.

EXAMPLE:

Filter=1;
AddTextColumn(WriteIf(Close>Ref(Close,-1),"Advancing","Declining"),"Condition");

Question: Why doesn't the formula display adequate text for bars when the condition is met?

Answer:  WriteIf( ) function returns a STRING, not an array of strings, so only the last value is displayed for all the bars.

  1. SetBarsRequired( )

Sets the number of previous and future bars needed for a formula to properly execute.

EXAMPLE 1:

The below formula should plot a % change indicator for the last 10 bars, but it behaves strangely:

LookbackPeriod = 10;
StartBar = (BarCount -1) - LookbackPeriod;
PrecChange =(Close/Close[StartBar]-1)*100;
PercentChangeGraph = IIf(BarIndex() >= StartBar,PrecChange ,Null);

Plot(PercentChangeGraph,"Close",colorRed,styleLine);

EXAMPLE 2:

The below formula should plot a DownArrow for the last bar, but it doesn’t. Why:

Buy= (BarCount-1) == BarIndex();
PlotShapes(Buy*shapeDownArrow,colorRed);

Due to plotting efficiency reasons, not all the bars are loaded into indicators. Some (not all) formulas require all bars to be loaded (here: BarCount value depends on the number of bars loaded). The solution for both these cases is to use the SetBarsRequired( ) function with a relatively high number of bars to load – the following line at the beginning of the formula will cause the necessary number of bars to be loaded:

SetBarsRequired(999999,999999);

  1. Using parentheses

Parentheses can be used to control the operation precedence (the order in which the operators are calculated). AmiBroker always performs operations within the innermost parentheses first. To learn the precedence of operations when parentheses are not used, visit: http://www.amibroker.com/guide/a_language.html

EXAMPLE:

“I would like to buy whenever either Close is higher than its 10-periods Moving Average or Close is at least 10% higher than yesterday's close, but the buy should only apply when Current Volume is higher than its 10-period Moving Average. However, I get Buy signals for the days when Volume is lower than MA(Volume,10). Why?”

Buy = Close > MA(Close,10) OR Close == 1.1*Ref(Close,-1) AND  

Volume > MA(Volume,10);

The solution is to add parentheses, otherwise, the system buys whenever Close > MA(Close,10) condition is met (or Close == 1.1*Ref(Close,-1) AND  Volume > MA(Volume,10) are both met).

Buy = ( Close > MA(Close,10) OR Close == 1.1*Ref(Close,-1) ) AND  

Volume > MA(Volume,10);
